# Backlog Generator Command

`/backlog` 명령어를 입력하면 애자일 백로그 아이템을 생성합니다.

---

⚠️ **중요: 출력 형식 규칙 (필수 준수)**

반드시 아래 마크다운 형식을 정확히 따를 것:
- **Purpose**: `**현재 상황**:`, `**이 작업을 통해**:`, `**기대 효과**:` → 볼드체 필수
- **Success Criteria**: `- [ ]` 체크박스 형식 필수
- **To-Do**: `- [ ]` 체크박스 형식 필수
- **파일명/커밋**: 백틱(`) 으로 감싸기 필수

---

## Instructions

1. **테스트 선택 확인**:
   - 사용자가 plan.md에서 텍스트 블록을 선택했으면 → 선택한 테스트에 대한 백로그 생성
   - 선택하지 않았으면 → plan.md를 읽어서 다음 체크되지 않은 테스트를 찾아 백로그 생성

2. **백로그 생성**: 아래 형식에 맞춰 백로그를 콘솔에 출력

3. **Markdown 형식**: 사용자가 복사해서 애자일 보드에 붙여넣을 수 있도록 마크다운 형식으로 출력

## 백로그 카드 구성 요소

백로그는 다음 항목들로 구성됩니다:

### 1. 제목 (Title)
- 테스트 이름에서 추출한 핵심 기능 설명
- **누구나 이해할 수 있는 언어**로 작성 (기술 용어 최소화)
- 형식: `[Domain] 명확하고 구체적인 기능 설명`
- 예시: `[User Domain] Google 로그인으로 사용자 정보 자동 저장`

### 2. Domain
- 어느 도메인에 속하는지 명시
- 가능한 값: `User Domain`, `Board Domain`, `Session & Authentication`, `Integration & E2E`

### 3. Purpose (목적)
- **왜 이 작업이 필요한지** 비즈니스 가치와 배경 설명
- prd.md의 요구사항과 연결
- 현재 문제점 또는 개선 필요성 명시
- **비개발자도 이해할 수 있도록** 작성
- 2-4문장으로 명확하게
- 예시 형식:
  - "현재 상황: ..."
  - "이 작업을 통해: ..."
  - "기대 효과: ..."

### 4. Success Criteria (완료 조건)
체크리스트 형식으로 작성:
- **기능적 완료 조건** 중심 (사용자 시나리오 기반)
- 비개발자(PM, QA)도 확인 가능한 수준
- 구체적이고 측정 가능한 조건
- TDD 단계 포함: 테스트 작성(RED), 구현(GREEN), 리팩터링(REFACTOR)
- 커밋 완료

### 5. To-Do (작업 목록)
TDD 사이클에 맞춘 작업 순서를 체크리스트 형식으로 작성:
- 테스트 작성 (`test_xxx.py`)
- 테스트 실행 및 실패 확인 (RED)
- 최소한의 구현 코드 작성
- 테스트 실행 및 통과 확인 (GREEN)
- 코드 리팩터링 (필요 시)
- 모든 테스트 실행 및 통과 확인
- 커밋: `<type>: [작성자] <요약> [AIS-XX]`

### 6. Issue / Blockers (Optional)
- 예상되는 문제점이나 의존성이 있을 경우에만 포함
- 없으면 생략 가능

## 출력 형식 예시

```markdown
# [User Domain] Google 로그인으로 사용자 정보 자동 저장

## Domain
User Domain

## Purpose (목적)
**현재 상황**: 시스템이 Google OAuth2 인증을 지원하지만, 로그인한 사용자의 정보를 저장하고 관리할 수 있는 기능이 없습니다.

**이 작업을 통해**: Google 계정으로 로그인하면 사용자의 기본 정보(이메일, 이름, 프로필 사진)가 자동으로 시스템에 저장됩니다.

**기대 효과**: 사용자는 별도의 회원가입 절차 없이 Google 계정 하나로 즉시 서비스를 이용할 수 있고, 시스템은 각 사용자를 식별하여 맞춤형 서비스를 제공할 수 있습니다. 이는 향후 게시판 작성자 관리, AI 에이전트 개인화 등의 기반이 됩니다.

## Success Criteria (완료 조건)
- [ ] Google 로그인한 사용자의 정보(Google ID, 이메일, 이름, 프로필 사진)가 시스템에 자동으로 저장됨
- [ ] 동일한 Google 계정으로 재로그인하면 기존 저장된 정보를 조회할 수 있음
- [ ] 저장된 정보가 Google에서 제공한 원본 정보와 일치함

## To-Do (작업 목록)
- [ ] `test_user_creation_with_google_info` 테스트 작성 (`tests/user/domain/test_user.py`)
- [ ] 테스트 실행 및 실패 확인 (RED)
- [ ] User 엔티티 클래스 작성 (`user/domain/user.py`)
- [ ] Google 정보를 저장하는 생성자 구현 (google_id, email, name, profile_picture)
- [ ] 테스트 실행 및 통과 확인 (GREEN)
- [ ] 코드 품질 검토 및 리팩터링 (필요 시)
- [ ] 전체 테스트 스위트 실행
```

## 작성 가이드

### Notion 에자일 보드를 사용 예정
- 노션에 붙여넣기 쉬운 형식으로 만들것

### Title (제목) 작성 시
- **비개발자도 이해 가능**: "User 엔티티 생성" 대신 "사용자 정보 자동 저장"
- **기술 용어 최소화**: "OAuth2 토큰 발급" 대신 "Google 로그인"
- **구체적인 행동**: "구현", "개발" 같은 모호한 표현보다 "저장", "조회", "수정" 같은 명확한 동사 사용

### Purpose (목적) 작성 시
- **3단계 구조 사용**:
  1. **현재 상황**: 무엇이 부족한지, 어떤 문제가 있는지
  2. **이 작업을 통해**: 구체적으로 무엇을 하는지
  3. **기대 효과**: 사용자/시스템에게 어떤 가치를 주는지
- **prd.md 연결**: 어떤 요구사항을 충족하는지 명시
- **비즈니스 가치 포함**: 기술적 내용보다 비즈니스/사용자 가치 중심

### Success Criteria (완료 조건) 작성 시
- **기능 중심**: "코드 작성" 보다는 "~하면 ~된다" 형식의 기능 동작 조건
- **비개발자 확인 가능**: PM, QA가 직접 테스트할 수 있는 수준으로
- **구체적이고 측정 가능**: "잘 동작한다" 대신 "Google ID로 조회하면 저장된 정보가 반환된다"
- **TDD 단계 포함**: RED, GREEN, REFACTOR 단계 명시
- **명확한 조건**: 애매모호한 표현 지양

### To-Do (작업 목록) 작성 시
- **TDD 원칙 준수**: Red → Green → Refactor 순서 엄격히
- **구체적인 파일 경로**: 어떤 파일을 작성/수정하는지 명시
- **커밋 메시지 형식**: `<type>: [작성자] <요약> [AIS-XX]` 준수
